# M1.3 Decision — Tauri: **GO** (2026-07-03)

## Verdict

**GO — build VAULT on Tauri.** Three.js + `@react-three/postprocessing` Bloom
holds a vsync-locked 60 FPS at a full-4K canvas on the real target (R9700 /
WebKitGTK). Electron fallback stays on the shelf; the front-end remains
runtime-portable per PRD §3.3.

## Environment (the real box, per PRD gate requirement)

- Pop!_OS 24.04, kernel 6.18.36-r9700-smufix, Wayland
- **WebKitGTK 2.52.3** (≥ 2.48 constraint satisfied; Skia GPU + WebGL GPU process era)
- Tauri v2 (wry/webkit2gtk-4.1), Rust stable
- AMD Radeon AI PRO R9700 (gfx1201, radeonsi/RADV, Mesa 25.2.8)
- Launched with `WEBKIT_FEATURES=UseGPUProcessForWebGL`; DMABUF/compositing
  left ENABLED (never set the `WEBKIT_DISABLE_*` flags on AMD — PRD §3.5)

## The one gotcha (MUST carry into production)

This box routes GL/EGL to the **NVIDIA card by default** (`10_nvidia.json`
sorts before `50_mesa.json` in glvnd). Run 1 reproduced exactly the
NVIDIA-specific WebKitGTK breakage the research doc warned about. The fix —
required in the production launcher (M0.2 finalization), not just the spike:

```bash
__EGL_VENDOR_LIBRARY_FILENAMES=/usr/share/glvnd/egl_vendor.d/50_mesa.json
__GLX_VENDOR_LIBRARY_NAME=mesa
DRI_PRIME=1
```

The M1.2 empirical probe (frame-time measurement, not renderer string — which
WebKitGTK fakes as "Apple GPU"… actually reports a Safari UA here) correctly
flagged run 1 and passed run 2; keep it as a startup self-check in the shipped
app so a driver/glvnd regression is caught immediately.

## Consequences

- M3 (3D graph) unblocked — proceed on Tauri/R3F.
- M0.2 finalization unblocked — tray, hotkey, sidecar; launcher must pin AMD GL
  and keep the startup frame probe.
- Tauri#6559 (WebGL context lost) did NOT reproduce during the spike runs.
  Watch for it during longer sessions; probe stays in as the canary.
